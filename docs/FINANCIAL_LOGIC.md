# Финансовая логика

Описание ядра бизнес-логики системы: грейс-период, лимиты, платежи, бюджеты и капитализация.

---

## Правило 120 дней (Grace Period)

### Концепция

Сбербанк предоставляет льготный период по кредитной карте МИР7600. При совершении покупки в месяце **M** пользователь может рассчитаться до последнего дня месяца **M+3** без начисления процентов.

**Примеры:**
- Покупка в **январе** → дедлайн **30 апреля**
- Покупка в **мае** → дедлайн **31 августа**
- Покупка в **октябре** → дедлайн **31 января** (следующего года)
- Покупка в **декабре** → дедлайн **31 марта** (следующего года)

### Формула расчёта

```
дедлайн = первый_день_месяца_покупки + 4 месяца - 1 день
```

В коде:
```python
grace_deadline = date(month + 4 months) - timedelta(days=1)
```

Пример кода в `services/credit_logic.py::CreditCardService.calculate_grace_deadline()`.

### Отчётные периоды

Каждый календарный месяц, в котором были траты по кредитной карте, создаёт один **отчётный период** (`BillingPeriod`):

| Поле | Значение |
|------|---------|
| `month` | 1-е число месяца (ключ для группировки) |
| `grace_deadline` | Дедлайн рассчётов по грейсу |
| `total_spent` | Сумма всех трат в месяц |
| `is_closed` | Закрыт ли период (долг погашен) |

### Использование в API

Список открытых периодов доступен в:
- `GET /api/finance/summary` (поле `open_periods`)
- `GET /api/finance/periods` (полный список)

## Контроль кредитного лимита (Hard Limit)

### Определение

**Hard Limit** — это жёсткое ограничение на сумму, которую пользователь может потратить в один момент. Если сумма транзакции превышает доступный лимит, операция **отклоняется** с ошибкой.

### Формула

```
доступный_лимит = кредитный_лимит - сумма_всех_неоплаченных_трат
```

**Где:**
- `кредитный_лимит` — полный лимит карты (по умолчанию 150 000 ₽)
- `сумма_всех_неоплаченных_трат` — сумма всех транзакций в открытых отчётных периодах

### Пример

```
Кредитный лимит: 150 000 ₽
Траты в апреле (открытый период): 55 000 ₽
Траты в марте (открытый период): 30 000 ₽
Всего неоплачено: 85 000 ₽

Доступный лимит = 150 000 - 85 000 = 65 000 ₽
```

Если пользователь попытается потратить 70 000 ₽, система отклонит операцию и вернёт ошибку.

### Проверка в коде

Реализация в `services/credit_logic.py::CreditCardService._process_credit_transaction()`:

```python
available_limit = await CreditCardService.get_available_limit(db)
if amount > available_limit:
    return {
        "success": False,
        "errors": [f"Превышен доступный лимит. Доступно: {available_limit} ₽"]
    }
```

### Настройка

Кредитный лимит задаётся переменной окружения `CREDIT_LIMIT` (по умолчанию 150000).

## Обработка платежей (FIFO)

### Концепция

Когда пользователь платит по кредитной карте, платёж должен быть применён к наиболее срочному (старому) открытому отчётному периоду. Используется алгоритм **FIFO** (First-In-First-Out).

### Логика

1. **Найти открытые периоды** — запросить все `BillingPeriod` где `is_closed = False`
2. **Сортировать по дедлайну** — упорядочить по `grace_deadline` (восходящий порядок)
3. **Выбрать первый** — применить платёж к периоду с самым ранним дедлайном
4. **Создать запись платежа** — добавить `CreditPayment` запись с суммой
5. **Проверить закрытие** — если сумма всех платежей ≥ сумме трат за период → закрыть период

### Пример

```
Открытые периоды:
1. Апрель 2026: traты=55 000 ₽, дедлайн=31 июля, платежи=0 ₽
2. Март 2026: траты=30 000 ₽, дедлайн=30 июня, платежи=10 000 ₽ (уже частично оплачен)

Порядок применения платежа:
1. Сортируем по дедлайну: Март (30 июня) ← Апрель (31 июля)
2. Применяем платёж к Марту (самый срочный)
```

### Автоматическое закрытие периода

```
Март: траты=30 000 ₽, текущие платежи=10 000 ₽
Новый платёж: 20 000 ₽
Всего платежей = 10 000 + 20 000 = 30 000 ₽

30 000 ≥ 30 000? → ДА
Результат: период закрыается (is_closed = True)
```

### Реализация в коде

Метод `services/credit_logic.py::CreditCardService._process_payment()`:

```python
# Получить открытые периоды, отсортированные по дедлайну (FIFO)
open_periods = await AsyncORM.get_open_billing_periods(db)

# Выбрать первый (самый срочный)
if not open_periods:
    return success with warning

period = open_periods[0]

# Создать запись платежа
await AsyncORM.create_credit_payment(db, period.id, amount)

# Сумировать все платежи за период
total_paid = await AsyncORM.get_period_total_payments(db, period.id)

# Проверить закрытие
if total_paid >= period.total_spent:
    await AsyncORM.close_billing_period(db, period.id)
```

## Категорийные лимиты дебетовой карты (Soft Limit)

### Концепция

По дебетовой карте ECMC6517 пользователь может установить месячный лимит по категориям расходов (Продукты, Транспорт и т.д.). Это **Soft Limit** — мягкое ограничение:

- Если потрачено **больше лимита** → система выдаёт **предупреждение** (warning)
- Но операция **НЕ блокируется** и выполняется нормально

### Использование

#### Создание/обновление бюджета

```bash
POST /api/finance/budgets/Продукты?limit=20000
```

Установить месячный лимит 20 000 ₽ на категорию "Продукты".

#### Получить список бюджетов

```bash
GET /api/finance/budgets
```

Вернёт список всех активных категорийных лимитов.

### Логика проверки

Когда приходит расход по дебетовой карте:

1. **Определить категорию** из результата парсинга Ollama (default: "Прочие")
2. **Найти лимит** для этой категории в таблице `budget_limits`
3. **Сумировать траты** за текущий месяц по этой категории
4. **Сравнить** с лимитом:
   - Если `текущие_траты + новая_операция > лимит` → добавить warning
   - В любом случае → операция выполняется

### Пример

```
Бюджет: Продукты = 20 000 ₽/месяц
Текущие траты в апреле: 18 000 ₽
Новая операция: 3 000 ₽
Сумма: 18 000 + 3 000 = 21 000 ₽

21 000 > 20 000? → ДА
Результат: 
  - Операция выполнена (21 000 ₽ добавлено в БД)
  - Возвращено warning: "Превышен лимит по категории Продукты на 1000 ₽"
  - Если настроена VK, отправлено уведомление с warning
```

### Реализация в коде

Метод `services/credit_logic.py::CreditCardService._process_debit_transaction()`:

```python
# Получить бюджет для категории
budget = await AsyncORM.get_budget_limit(db, category)

if budget:
    # Сумировать текущие траты за месяц
    spent_this_month = await AsyncORM.get_month_category_expenses(
        db, category, current_month
    )
    remaining = budget.monthly_limit - spent_this_month
    
    if amount > remaining:
        # Добавить warning, но не блокировать
        warnings.append(f"Превышен лимит по {category}: {amount} > {remaining}")

# Операция всегда выполняется (success: True)
```

## Grace-Unsafe операции

### Определение

Некоторые операции по кредитной карте **нарушают льготный период** и могут привести к начислению процентов, даже если долг рассчитан в дедлайн:

1. **Снятие наличных** — `account_type = credit, category = "Снятие наличных"`
2. **Переводы** — `category = "Перевод между счетами"`
3. **Комиссии** — `transaction_type = "fee"`

### Детектирование

При парсинге СМС в `routers/webhook.py` система проверяет:

```python
is_grace_safe = True
if parsed_result.account_type == "credit":
    if parsed_result.category in [
        "Снятие наличных",
        "Перевод между счетами",
        "Комиссия"
    ]:
        is_grace_safe = False
```

### Обработка

Когда обнаружена grace-unsafe операция:

1. **Логирование** — `CRITICAL` уровень логирования
2. **Алерт** — метод `GraceUnsafeAlarm.trigger()`
3. **Warning в API** — добавляется warning в ответ эндпоинта
4. **VK уведомление** — если настроено, отправляется критическое уведомление

### VK Уведомление

```
[ КРИТИЧЕСКОЕ ПРЕДУПРЕЖДЕНИЕ ]
Счет: 7600 (credit)
Сумма: 50000 руб.
Детали: ATM WITHDRAWAL (Снятие наличных)

ВНИМАНИЕ: Зафиксирована операция (снятие/перевод), 
нарушающая льготный период! Проценты по карте могут быть начислены.
```

### Пример в коде

`services/credit_logic.py::CreditCardService._process_credit_transaction()`:

```python
if not is_grace_safe:
    await GraceUnsafeAlarm.trigger(
        transaction_id=transaction.id,
        amount=amount,
        merchant=merchant,
        transaction_type=transaction_type
    )
    warnings.append(f"Grace-unsafe: {transaction_type} нарушает период")
```

## Капитализация накопительного счёта (Daily Yield)

### Концепция

По накопительному счёту (*1837) Сбербанк начисляет ежедневный процент. Система автоматически вычисляет дневной доход и сохраняет его в таблицу `daily_yields`.

### Формула

```
дневной_доход = баланс × (годовая_процентная_ставка / 100) / 365
```

Результат округляется до копеек (2 знака после запятой).

### Пример

```
Баланс счёта: 250 000 ₽
Процентная ставка: 11.5% (в апреле 2026)

Дневной доход = 250 000 × (11.5 / 100) / 365
              = 250 000 × 0.115 / 365
              = 78.77 ₽
```

### Таблица `daily_yields`

Каждый день в таблице сохраняется одна запись (или обновляется, если запись на день уже существует):

| Поле | Значение |
|------|---------|
| `date` | День расчёта (PRIMARY KEY) |
| `account_tail` | "1837" |
| `end_of_day_balance` | Баланс на конец дня |
| `applied_rate` | Процентная ставка (%) |
| `earned_amount` | Дневной доход |

### Процентные ставки (по месяцам)

В `config.py` определён словарь `SAVINGS_RATES`:

```python
SAVINGS_RATES = {
    "2026-04": 11.5,     # апрель 2026: 11.5% годовых
    "2026-05": 10.0,     # май 2026: 10.0% годовых
    # ...
    "default": 7.0,      # для остальных месяцев: 7.0% годовых
}
```

Ставка для текущего месяца получается вызовом `get_current_rate()`.

### Расчёт (Celery задача)

Каждый день в 23:55 Celery Beat запускает задачу `calculate_and_store_yield`:

1. **Получить последний баланс** — запрос последней транзакции по счету 1837 с `balance_after`
2. **Если баланса нет** → ничего не делать (return)
3. **Получить текущую ставку** → вызов `get_current_rate()`
4. **Вычислить доход** → применить формулу выше
5. **Сохранить/обновить** → запись в `daily_yields`
6. **VK дайджест** — если настроено, отправить дневной дайджест в мессенджер

### VK Дайджест

```
ФИНАНСОВЫЙ ДАЙДЖЕСТ: НАКОПИТЕЛЬНЫЙ СЧЕТ
Счет: *1837
Примененная ставка: 11.5% годовых

Доход за сегодня: +78.77 руб.
Текущий баланс: 250000 руб.
```

## Порядок обработки транзакций

Когда приходит СМС от банка, система выполняет следующие шаги:

### 1. Создание записи в БД

```
POST /api/sber-webhook
↓
AsyncORM.create_transaction(db, sms_text=...)
↓
В таблицу transactions добавляется запись с raw SMS
```

### 2. Парсинг СМС (Ollama)

```
ollama_client.parse_sms(sms_text)
↓
Ollama возвращает JSON: amount, type, merchant, category, is_expense, balance_after
↓
Если ошибка → status="parse_error", return
```

### 3. Обновление транзакции результатами

```
AsyncORM.update_transaction_parsed(db, transaction, ...)
↓
Сохраняются: card_tail, account_type, amount, category, is_grace_safe, и т.д.
```

### 4. Финансовая логика (основное)

```
CreditCardService.process_transaction(db, ...)
↓
Проверка account_type → routing:
  - "credit" → _process_credit_transaction()
  - "debit"  → _process_debit_transaction()
  - "savings" → skip (only log)
↓
Для credit: проверка лимита, создание/привязка BillingPeriod, проверка grace-safety
Для debit: проверка категорийного лимита, warning если превышен
↓
Возвращение результата: {success, billing_period, warnings, errors}
```

### 5. VK Уведомление (асинхронно)

```
Если настроена VK интеграция:
  - Формирование сообщения (обычное или критическое)
  - Добавление warnings/errors
  - asyncio.create_task(send_and_close())
  → сообщение отправляется в фоне
```

## Заметки и Edge Cases

### `savings_min_balance` не используется

Переменная `SAVINGS_MIN_BALANCE` определена в конфиге, но в текущей логике не применяется. Зарезервирована для будущей функции (например, блокировка операций если баланс < минимума).

### Дебетовые лимиты всегда мягкие

Даже если дебетовый лимит превышен, операция выполняется и сохраняется в БД. Только warning добавляется.

### Платежи всегда применяются к текущему дню

Метод `_process_payment()` использует `date.today()`, игнорируя временную метку СМС. Это означает, что платёж всегда связывается с текущей датой, не с датой из СМС.
