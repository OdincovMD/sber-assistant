# Sber Grace Assistant

Автоматизированная **локальная** система мониторинга финансов для реализации стратегии «Кредитная карусель» (120 дней без процентов от Сбербанка).

Система перехватывает СМС-уведомления от банка (через iOS Shortcuts), локально парсит их с помощью LLM (Ollama), и помогает пользователю максимизировать доход от остатка на накопительных счетах, не нарушая льготный период кредитной карты и отслеживая целевые траты для получения бонусов.

## Основные возможности

* **Интеллектуальный парсинг СМС:** Извлечение суммы, типа операции (покупка, перевод, снятие наличных, зачисление), имени мерчанта и остатка по карте с помощью локальной модели `qwen2.5:0.5b`. Формирование структурированного JSON.
* **Асинхронная обработка:** Webhook принимает СМС и мгновенно возвращает ответ (`status: "queued"`). Парсинг LLM, финансовая логика и уведомления выполняются в фоне через Celery — никаких таймаутов на стороне клиента.
* **Правило 120 дней:** Точный автоматический расчет дедлайна льготного периода Сбера (месяц покупки + 3 следующих полных месяца).
* **Мониторинг лимита и бонусов:** Автоматический пересчет доступного кредитного лимита и отслеживание выполнения целевых трат (по умолчанию 100 000 ₽) для бонуса к проценту по накопительному счету.
* **Категорийные бюджеты:** Мягкие лимиты расходов по категориям для дебетовой карты с еженедельным VK-отчётом по воскресеньям.
* **AI-аналитик:** Каждую пятницу в 19:00 локальный LLM анализирует финансовую сводку и генерирует 3 персонализированных рекомендации, которые отправляются в VK в формате банковского отчёта.
* **Анализ Grace-Unsafe операций:** Детектирование штрафных операций (перевод кредитных средств, снятие наличных) с немедленным логированием `CRITICAL` алерта и VK-уведомлением.
* **Инвестиционный портфель:** Учёт БПИФ и открытых ПИФ, P&L по каждому лоту, ЛДВ-календарь (налоговая льгота 3 года), анализ ликвидности портфеля для погашения грейса, сравнение SBMM vs накопительный счёт.
* **ЦБ РФ интеграция:** Ключевая ставка и RUONIA в реальном времени через официальный SOAP API ЦБ, расписание заседаний Совета директоров.
* **Уведомления в VK:** Мгновенное уведомление по каждой операции, ежедневный дайджест по накопительному счёту, еженедельный отчёт по бюджетам, пятничный AI-совет, ЛДВ-алерты.
* **Полная приватность:** Все вычисления, база данных и искусственный интеллект (Ollama) работают исключительно локально через Docker.

## Архитектура (Docker Compose)

Проект состоит из микросервисов:
* **Nginx (Inbound прокси):** Принимает входящие webhook-и от iOS на порту 80 и проксирует их.
* **FastAPI (Backend):** Ядро бизнес-логики, эндпоинты, процессинг транзакций и связь с БД.
* **PostgreSQL:** Надежное хранение оригинальных СМС, результатов парсинга и финансовых расчетов.
* **Ollama (LLM):** Локальный сервер с моделью для обработки естественного языка.
* **Redis:** Брокер сообщений и слой кэширования для фоновых задач.
* **Celery Worker & Beat:** Фоновый парсинг СМС через LLM, расчёт ежедневной капитализации, еженедельные бюджетные отчёты и пятничный AI-анализ.

## Быстрый старт

### 1. Подготовка окружения
Скопируйте пример файла конфигурации и заполните пароль к БД:
```bash
cp .env.example .env
# Отредактируйте .env (POSTGRES_PASSWORD и порты при необходимости)
```

### 2. Запуск инфраструктуры
Для управления контейнерами используйте `Makefile`:
```bash
# Поднять все контейнеры в фоне
make up

# Остановить инфраструктуру
make down

# Полностью пересобрать образы (полезно при изменении Python-кода)
make rebuild
```

### 3. Загрузка LLM модели
При первом запуске необходимо скачать модель в Ollama (занимает время, зависит от скорости интернета):
```bash
make ollama-pull
```

### 4. Проверка работоспособности
```bash
# Проверить статус всех компонентов (должно везде быть "ok")
make health

# Отправить тестовый вебхук напрямую
make test-webhook
```

#### `POST /api/sber-webhook`

**Запрос:**
```json
{
  "sms_text": "Покупка 5000р ПЯТЕРОЧКА. Баланс: 50000р"
}
```

**Ответ:**
```json
{
  "status": "queued",
  "transaction_id": 123
}
```

**Что происходит:**
1. СМС сохраняется в БД, ответ возвращается немедленно (< 1 сек)
2. В фоне Celery запускает `process_sms_task(transaction_id)`:
   - СМС отправляется в Ollama для парсинга (извлечение суммы, типа, мерчанта, баланса)
   - Результаты парсинга сохраняются в транзакцию
   - Запускается финансовая логика (привязка к отчётному периоду, проверка лимитов, грейс-периода)
   - Если настроена VK интеграция, отправляется уведомление в ВКонтакте

## Финансовая логика

### Правило 120 дней (Grace Period)

Когда пользователь совершает покупку в месяце **M**, льготный период заканчивается в **последний день месяца M+3**:
- Покупка в январе → дедлайн 30 апреля
- Покупка в мае → дедлайн 31 августа
- Покупка в декабре → дедлайн 31 марта следующего года

Формула: `дедлайн = первый день месяца покупки + 4 месяца - 1 день`

Система автоматически вычисляет дедлайн для каждого отчётного периода и отслеживает его в эндпоинте `/api/finance/periods`.

### Контроль кредитного лимита (Hard Limit)

Доступный лимит вычисляется как:
```
доступный_лимит = кредитный_лимит - сумма_всех_неоплаченных_трат
```

По умолчанию кредитный лимит установлен на **150 000 ₽** (для карты МИР7600). Если покупка превысит доступный лимит, система отклонит её с ошибкой. Это **жёсткое** ограничение.

### Обработка платежей (FIFO)

Когда в системе регистрируется платёж:
1. Система находит самый старый **открытый** отчётный период (по дедлайну льготы)
2. Создаёт запись `CreditPayment`, связанную с этим периодом
3. Суммирует все платежи по периоду
4. Если сумма платежей ≥ сумме трат за период → период автоматически закрывается (`is_closed = True`)

Таким образом, платежи применяются в порядке FIFO (First-In-First-Out) к наиболее срочным периодам.

### Категорийные лимиты дебетовой карты (Soft Limit)

По дебетовой карте можно установить месячный лимит по категориям (например, Продукты, Транспорт). Это достигается через:
- `POST /api/finance/budgets/Продукты?limit=20000` — установить лимит 20 000 ₽ на категорию

Если в месяце по категории потрачено больше, чем лимит, система выдаёт **предупреждение** (warning), но операцию **не блокирует**. Это **мягкое** ограничение.

### Grace-Unsafe операции

Некоторые операции нарушают льготный период и могут привести к начислению процентов:
- **Снятие наличных** с кредитной карты
- **Переводы** кредитных средств между счётами
- **Комиссии** банка (снятие, обслуживание)

Когда система обнаруживает такую операцию:
1. Логирует `CRITICAL` алерт в логи
2. Добавляет предупреждение в ответ эндпоинта
3. Если настроена VK интеграция, отправляет критическое уведомление в мессенджер

### Капитализация накопительного счёта (Daily Yield)

Каждый день в 23:55 Celery задача вычисляет начисленный доход по накопительному счёту:
```
дневной_доход = баланс * (процентная_ставка / 100) / 365
```

Результат округляется до копеек и сохраняется в таблицу `daily_yields`. Процентная ставка зависит от месяца (см. раздел Конфигурация).

## Планировщик задач (Celery Beat)

В системе работают шесть фоновых задач через Celery Beat и Worker.

### Ежедневный расчёт доходов (`calculate_and_store_yield`)

**Расписание:** Каждый день в **23:55 Московского времени**

**Логика:**
1. Получает последнюю транзакцию по накопительному счёту с заполненным балансом
2. Если такой нет → ничего не делает
3. Получает текущую процентную ставку (зависит от месяца)
4. Вычисляет дневной доход по формуле: `баланс * (ставка / 100) / 365`
5. Сохраняет запись в таблицу `daily_yields` (или обновляет, если уже есть запись на сегодня)
6. Если настроена VK интеграция, отправляет дневной дайджест

**Формат VK сообщения:**
```
ФИНАНСОВЫЙ ДАЙДЖЕСТ: НАКОПИТЕЛЬНЫЙ СЧЕТ
Счет: *1837
Примененная ставка: 11.5% годовых

Доход за сегодня: +95.20 руб.
Текущий баланс: 250000 руб.
```

### Еженедельный отчёт по бюджетам (`send_weekly_budget_report`)

**Расписание:** Каждое **воскресенье в 20:00 Московского времени**

**Логика:**
1. Получает все активные категорийные лимиты (таблица `budget_limits`)
2. Для каждой категории вычисляет траты в текущем месяце
3. Вычисляет остаток (лимит - траты)
4. Строит текстовый отчёт и отправляет в VK (если настроено)

**Формат VK сообщения:**
```
BUDGET REPORT (CURRENT MONTH)

Category: Продукты
Limit: 20000 RUB
Spent: 8500 RUB
Remaining: 11500 RUB
-----------------------------------
Category: Транспорт
Limit: 10000 RUB
Spent: 3200 RUB
Remaining: 6800 RUB
-----------------------------------
```

### Парсинг СМС (`process_sms_task`)

**Расписание:** Триггер по событию — запускается webhook-ом при получении СМС

**Логика:**
1. Извлекает сырой текст СМС из БД по `transaction_id`
2. Отправляет в Ollama для парсинга (LLM извлекает сумму, тип, мерчанта, баланс, тип счёта)
3. Сохраняет результат парсинга в транзакцию
4. Вызывает финансовую логику (`CreditCardService.process_transaction`)
5. Если операция grace-unsafe — формирует критическое уведомление
6. Отправляет банковское уведомление в VK с актуальными лимитами и прогрессом бонуса

### Получение цен MOEX (`fetch_investment_prices`)

**Расписание:** Каждый день в **19:05 Московского времени** (после закрытия торгов MOEX в 18:50)

**Логика:**
1. Запрашивает текущие цены биржевых БПИФ (SBGB, SBGD, SBFR, SBMM) через MOEX ISS API
2. Сохраняет в таблицу `investment_prices` (upsert по дате)
3. ПИФ Накопительный (PIF_NAK) обновляется вручную через `POST /api/investment/price`

### ЛДВ-алерты (`check_ldv_alerts`)

**Расписание:** Каждый день в **08:00 Московского времени**

**Логика:**
1. Проверяет все активные лоты портфеля
2. Алерт отправляется при: 90 дней до ЛДВ, 30 дней до ЛДВ, в день наступления ЛДВ
3. Если настроена VK интеграция, отправляет сообщение с расчётом потенциального налога

**Формат VK сообщения:**
```
[ ЛДВ УВЕДОМЛЕНИЕ ]
SBGB: 30 дн. до ЛДВ — НЕ продавать! Потенциальный налог при продаже: 3 500 ₽
```

### AI-советник (`send_ai_advice`)

**Расписание:** Каждую **пятницу в 19:00 Московского времени**

**Логика:**
1. Получает полную финансовую сводку (`CreditCardService.get_financial_summary`)
2. Отправляет JSON-сводку в Ollama с профессиональным system-промптом
3. LLM генерирует ровно 3 конкретных совета на русском языке (без эмодзи)
4. Отправляет результат в VK с заголовком `[ СОВЕТ ОТ ИИ-АНАЛИТИКА ]`

**Формат VK сообщения:**
```
[ СОВЕТ ОТ ИИ-АНАЛИТИКА ]

1. Текущий доступный лимит составляет 144 000 руб. Рекомендуется воздержаться
   от крупных покупок до закрытия периода апреля.
2. До достижения цели 100 000 руб. осталось 94 000 руб. Увеличьте безналичные
   расходы по кредитной карте для получения бонуса.
3. Льготный период истекает 31 июля. Обеспечьте полное погашение задолженности
   не позднее 25 июля.
```

## База данных

Все данные хранятся в PostgreSQL. Схема включает 5 основных таблиц.

### `transactions` — Таблица транзакций

Каждая СМС от банка создаёт запись в этой таблице.

| Поле | Тип | Описание |
|------|-----|---------|
| `id` | INTEGER PK | Уникальный идентификатор |
| `billing_period_id` | FK | Привязка к отчётному периоду (может быть NULL) |
| `sms_text` | TEXT | Оригинальный текст СМС |
| `card_tail` | VARCHAR(4) | Последние 4 цифры карты (7600, 6517, 1837) |
| `account_type` | ENUM | Тип счёта: `credit` / `debit` / `savings` |
| `amount` | NUMERIC(12,2) | Сумма операции в рублях |
| `transaction_type` | ENUM | Тип: `purchase` / `payment` / `transfer` / `deposit` / `withdrawal` / `fee` / `unknown` |
| `merchant` | VARCHAR(255) | Название мерчанта (магазина, сервиса) |
| `category` | VARCHAR(100) | Категория расходов (Продукты, Транспорт, и т.д.) |
| `is_grace_safe` | BOOLEAN | Нарушает ли операция льготный период |
| `is_expense` | BOOLEAN | Расход (TRUE) или доход (FALSE) |
| `balance_after` | NUMERIC(12,2) | Баланс счёта после операции |
| `card` | VARCHAR(50) | Маска карты (ECMC6517, МИР7600, и т.д.) |
| `grace_deadline` | DATE | Дедлайн льготного периода (кэш из billing_period) |
| `raw_llm_response` | TEXT | Сырой ответ от Ollama (JSON) |
| `is_parsed` | BOOLEAN | Успешно ли парсилась СМС |
| `created_at` | TIMESTAMP | Время создания записи |

### `billing_periods` — Таблица отчётных периодов

Каждый календарный месяц, в котором были траты, имеет один отчётный период.

| Поле | Тип | Описание |

| `month` | DATE UNIQUE | Первый день месяца (ключ для группировки) |

| `grace_deadline` | DATE | Дедлайн льготного периода (месяц + 3 месяца) |
| `is_closed` | BOOLEAN | Закрыт ли период (все долги погашены) |
| `created_at` | TIMESTAMP | Время создания |

### `daily_yields` — Таблица дневных доходов по накопительному счёту

| `earned_amount` | NUMERIC(12,2) | Рассчитанный дневной доход |

### `budget_limits` — Таблица категорийных лимитов

## Настройка iOS Shortcuts (Команды)

Для автоматической передачи поступлений со смартфона:
1. Создайте автоматизацию: **"При получении сообщения"**. Отправитель — `900`. Запуск — "Немедленно".
2. Добавьте действие **"Получить содержимое URL"**.
3. Настройте URL: `http://<ВАШ_IP>/api/sber-webhook` (если доступен извне).
4. Метод `POST`, Заголовки (Headers): `Content-Type: application/json`.
5. Тело запроса (JSON): добавьте ключ `sms_text` и выберите Текст сообщения из переменных ярлыка.
